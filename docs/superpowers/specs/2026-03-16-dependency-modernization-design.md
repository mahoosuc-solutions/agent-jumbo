# Dependency Modernization Design

**Date:** 2026-03-16
**Status:** Approved
**Goal:** Resolve 37 known CVEs across 11 packages, modernize to latest stable versions, adopt `uv` as package manager, consolidate to single source of truth in `pyproject.toml`.

## Context

Four commits (RC2, RC3, Python 3.11 upgrade) just landed on `main`. The pre-commit `pip-audit` hook now blocks all commits due to 37 CVEs in 11 third-party packages. Dependencies are declared in both `requirements.txt` (exact pins) and `pyproject.toml` (floor ranges), and the two contradict each other. Additionally, 25 packages in `requirements.txt` are missing from `pyproject.toml` entirely.

## Decision: Three-Wave Upgrade

Split upgrades into three independently shippable PRs, tiered by risk. Each wave is tested and merged before the next begins.

| Wave | Packages | CVEs Fixed | Risk | Effort |
|------|----------|-----------|------|--------|
| 1 | pypdf, simpleeval, flask, aiohttp, lxml-html-clean, markdown + infra | 28 | Low | ~1 hr |
| 2 | fastmcp 2->3, unstructured 0.16->0.21 | 4 | Medium | ~1-2 hr |
| 3 | langchain-core 0.3->1.x, langchain-community 0.3->0.4.x | 5 | High | ~2-4 hr |

### Update `DockerfileLocal`

`DockerfileLocal` COPYs `requirements.txt` and `requirements2.txt` and installs from them. After deleting these files, the Dockerfile must be updated:

1. Remove `COPY ./requirements.txt` and `COPY ./requirements2.txt` lines
2. Replace `uv pip install -r requirements.txt` / `-r requirements2.txt` with `uv pip install -e .` (install from `pyproject.toml`)
3. Keep `requirements.lock.txt` path as an optimization layer (generated by `uv export`)
4. Keep `docker/wheelhouse` path for offline installs

### Adopt `uv`

- `uv lock` generates `uv.lock` (committed to git) for reproducible installs
- `uv sync` replaces `pip install -r requirements.txt`
- `uv venv` replaces `python -m venv`
- Any CI scripts or docs referencing `pip install` updated to `uv sync`

### Update pre-commit `pip-audit` hook

The current `.pre-commit-config.yaml` pip-audit hook uses `args: ["-r", "requirements.txt"]`. Since `requirements.txt` is being deleted, this must become a local hook:

```yaml
- repo: local
  hooks:
    - id: pip-audit
      name: pip-audit
      language: system
      entry: bash -c 'uv export --format requirements-txt --no-hashes | pip-audit --stdin --progress-spinner=off'
      pass_filenames: false
      files: '(pyproject\.toml|uv\.lock)$'
```

This runs `uv export` to generate a requirements-format stream and pipes it to `pip-audit --stdin`.

## Wave 1 -- Safe Bumps (28 CVEs)

### Version Changes

| Package | From | To | CVEs | Notes |
|---------|------|----|------|-------|
| `pypdf` | `>=6.0.0` | `>=6.8.0` | 14 | All security patches within 6.x |
| `simpleeval` | `==1.0.3` | `>=1.0.5` | 1 | Patch bump, no API changes |
| `flask[async]` | `>=3.0.3` | `>=3.1.3` | 1 | Minor bump; drops Python 3.8 (we require 3.11) |
| `aiohttp` | `>=3.9.0` | `>=3.13.3` | 8 | Cookie jar internals changed; timeout error types split |
| `lxml-html-clean` | `==0.3.1` | `>=0.4.4` | 3 | Transitive dep, no direct imports |
| `markdown` | `>=3.7` | `>=3.8.1` | 1 | Transitive dep, no direct imports |

### Validation Required

- **aiohttp timeout handling:** 7 files import aiohttp. All currently catch generic `TimeoutError` (Python built-in), not `aiohttp.ServerTimeoutError`. The 3.13.x timeout error type split should not break existing code. Quick grep to confirm.
- **aiohttp module-level sessions:** 3.13.x raises `RuntimeError` if `TCPConnector` or `ClientSession` is created outside a running event loop. Check all 7 files for module-level session creation (should be none â€” all create sessions inside async functions).
- **aiohttp cookie jars:** 3.13.x changed `filter_cookies()` domain/path matching. Verify none of the 7 files use persistent cookie jars (unlikely â€” all create ephemeral sessions).
- **Flask Werkzeug minimum:** Flask 3.1.x requires Werkzeug >= 3.1. Verify no direct Werkzeug imports depend on pre-3.1 APIs.

### Test Plan

- Full pytest suite, ruff lint, TypeScript type-check, Next.js build
- Docker build: `docker build -f DockerfileLocal .` to verify updated Dockerfile works

### Pass Criteria

All tests green, no new import errors, `uv lock` resolves cleanly, `pip-audit` shows 0 high/critical CVEs for Wave 1 packages, Docker image builds.

## Wave 3 -- Langchain Ecosystem (5 CVEs)

### Version Changes

| Package | From | To | Notes |
|---------|------|----|-------|
| `langchain-core` | `==0.3.49` | `>=1.2.19` | Major version jump |
| `langchain-community` | `==0.3.19` | `>=0.4.1` | Tracks core 1.x (community does NOT have 1.x releases) |
| `langchain-text-splitters` | `==0.3.7` (transitive) | No explicit pin | Transitive via langchain-community; verify resolves to compatible version |

**Important:** `langchain-community` uses its own versioning (0.4.x) to track `langchain-core` 1.x. There is no `langchain-community` 1.x on PyPI.

### Files Affected (11)

**LLM orchestration:**
- `agent.py` -- BaseMessage, SystemMessage, HumanMessage, AIMessage
- `models.py` -- ChatModel interfaces
- `python/helpers/call_llm.py` -- prompt templates (has try/except fallback from `langchain.prompts` to `langchain_core.prompts`)
- `python/helpers/history.py` -- message history types

**Document pipeline:**
- `python/helpers/knowledge_import.py` -- PyPDFLoader, TextLoader, CSVLoader
- `python/helpers/document_query.py` -- `from langchain.schema import SystemMessage, HumanMessage` (legacy), RecursiveCharacterTextSplitter (legacy import path)

**Vector DB / Memory (critical legacy imports):**
- `python/helpers/memory.py` -- `from langchain.embeddings import CacheBackedEmbeddings`, `from langchain.storage import InMemoryByteStore, LocalFileStore`
- `python/helpers/vector_db.py` -- `from langchain.embeddings import CacheBackedEmbeddings`, `from langchain.storage import InMemoryByteStore`
- `python/helpers/memory_consolidation.py` -- Document processing
- `python/api/memory_dashboard.py` -- memory API

### Known Legacy Import Paths Requiring Manual Migration

These imports go through the `langchain` meta-package and will break under 1.x:

| File | Import | New Path (1.x) |
|------|--------|----------------|
| `memory.py:11` | `from langchain.embeddings import CacheBackedEmbeddings` | `from langchain.embeddings import CacheBackedEmbeddings` (verify; may move to `langchain_community.embeddings`) |
| `memory.py:12` | `from langchain.storage import InMemoryByteStore, LocalFileStore` | `from langchain_core.stores import InMemoryByteStore` + `from langchain_community.storage import LocalFileStore` (or `langchain_classic`) |
| `vector_db.py:7-8` | Same `langchain.embeddings` / `langchain.storage` imports | Same fix as memory.py |
| `document_query.py:23` | `from langchain.schema import SystemMessage, HumanMessage` | `from langchain_core.messages import SystemMessage, HumanMessage` |
| `call_llm.py:5` | `from langchain.prompts import ChatPromptTemplate, ...` (try/except) | `from langchain_core.prompts import ChatPromptTemplate, ...` (remove try/except) |

**Note:** If `CacheBackedEmbeddings` or stores moved to `langchain_classic` (a separate package), we may need to add `langchain-classic` as a dependency OR find the `langchain_core`/`langchain_community` equivalents.

### Migration Strategy

1. **Keep `langchain` meta-package installed** during migration (needed for `langchain-cli migrate` to resolve existing imports)
2. **Automated pass:** Run `langchain-cli migrate --diff .` to preview, then `langchain-cli migrate .` for mechanical import renames and `pydantic_v1` -> `pydantic` migration
3. **Manual fixes** for imports the CLI misses (see table above), especially:
   - `langchain.embeddings.CacheBackedEmbeddings`
   - `langchain.storage.InMemoryByteStore` / `LocalFileStore`
   - `langchain.schema.SystemMessage` / `HumanMessage`
   - Remove try/except fallback in `call_llm.py` (use direct `langchain_core` imports)
4. **Pydantic v2 check:** Already on `pydantic>=2.11.7`. Verify no code uses `langchain_core.pydantic_v1` shims.
5. **Remove `langchain` meta-package** AFTER migration CLI has run and manual fixes are complete
6. **Verify `langchain-text-splitters`** resolves to a compatible version automatically (do not pin explicitly unless forced)

**Reference:** [LangChain v1 migration guide](https://docs.langchain.com/oss/python/migrate/langchain-v1)

### Test Plan

- Full pytest suite
- Integration: send message through agent pipeline end-to-end
- Document ingestion: PDF, HTML, CSV, plain text
- Vector search and memory retrieval
- Memory consolidation
- Docker build + test inside container (Python 3.12 in Docker vs 3.11 local)

### Pass Criteria

Full test suite green, agent processes messages, document pipeline ingests all formats, vector search returns results, Docker image builds and runs.

## Execution Order

```
Wave 1 (branch: deps/wave-1-safe-bumps)
  |- Reconcile requirements.txt + requirements2.txt into pyproject.toml
  |- Delete requirements.txt and requirements2.txt
  |- Update DockerfileLocal to install from pyproject.toml
  |- Install uv, generate uv.lock
  |- Bump 6 packages in pyproject.toml
  |- Update pre-commit pip-audit to local hook with uv export
  |- Test (pytest + Docker build), merge to main
  |
Wave 2 (branch: deps/wave-2-fastmcp-unstructured)
  |- Bump fastmcp 2->3, migrate mcp_server.py
  |- Bump unstructured + companions
  |- Test, merge to main
  |
Wave 3 (branch: deps/wave-3-langchain-1x)
  |- Install langchain-cli
  |- Run langchain-cli migrate (with langchain meta-package still installed)
  |- Manual fixes for legacy imports (embeddings, storage, schema, prompts)
  |- Remove langchain meta-package
  |- Test full pipeline + Docker build
  |- Merge to main
```
